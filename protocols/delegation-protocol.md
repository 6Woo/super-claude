# 위임 프로토콜 (상세)

메인(상위 모델)은 판단·분해·검증·통합만. 노동은 레인별로 위임. 설계는 외부 추론기 plan(계획)·challenge(반박)·review(리뷰) 교차검토로 검증했다.

## 대전제 (반박에서 걸러진 것)

- **"외부 추론기 공짜니까 막 밀어라"는 틀렸다.** 요금은 아껴도 메인의 맥락포장·검증·통합 비용은 남는다. 추론기는 stateless라 매 호출 repo규칙·금지파일·라이브위험·최근변경 재주입 필요 → 작은/중간 작업은 이 비용이 절감을 먹는다. **위임 문턱 = 덩어리 크고 독립적일 때만.**
- **병렬 2배는 조건부.** 독립성·안정 인터페이스일 때만 성립. 라이브 프로덕션은 DB/인증/배포/동시성이 얽혀 병렬 구현 분해 자체가 위험. 2배는 조사·분석·리뷰 국면에서 실현, 구현 국면은 신중.
- **"검증 후 통합" ≠ 품질보증.** 두 팀이 둘 다 그럴듯하게 틀리면 더 설득력 있는 오답을 고를 수 있음. 같은 맥락 공유 시 상관오류(같은 함정에 같이 빠짐).
- **규율은 프롬프트 문장이 아니라 도구/권한/워크플로가 강제한다.** 모델은 "읽었음/검증했음"을 말로만 충족한 척 가능.

## 3 레인

### 레인 A — 자동·공격적 OK (쓰기 0, 리스크 0)
탐색·구조조사·리서치·코드/설계 리뷰·반박·로그 요약·후보파일 나열. 저비용 모델 + 외부 추론기 병렬로 맘껏. 토큰절약·생산성·에이전트활용이 안전하게 사는 곳.
저비용 모델은 **조사보조원으로 못박기**: 파일찾기·검색·짧은 diff리뷰·로그분류·요약. **구현·다단계 리팩터·장애결론·프로덕션 판단 금지.**

### 레인 B — opt-in·가드 필수 (쓰기)
실제 구현/버그픽스. 중급 모델(좁은 파일소유권) + 추론기 초안. **자동 기본값 아님.** 병렬 구현은 ①worktree 격리 ②파일소유권 분리 ③테스트 존재, 3조건 모두일 때만. 얽힌 프로덕션은 순차. 안전은 구조적 강제(worktree, read-only 권한, 미션에 쓰기 허용/금지 명시). 같은 워킹트리 동시 구현 1개.

### 레인 C — 상위 모델 전용 (판단)
아키텍처, 라이브/DB/인증/보안/마이그레이션, 다중서비스 장애원인, 불확실 요구정리, 하위 산출물의 고위험 최종리뷰. 하위 모델로 안 내림.

## 하위모델 규율 코어 10줄 (위임 미션에 동봉)

1. 파일 안 읽고 그 내용 단정 금지. 안 읽었으면 읽는다.
2. 추측보다 증거: 검색/로그/테스트 출력 먼저.
3. 수정 전 영향받는 파일 먼저 식별.
4. 작은 단계로. 요청 범위 밖 리팩터·기능추가·스타일변경 금지.
5. 수정 후 검증(테스트/빌드/타입체크) 실행하거나 명령 제시.
6. 존재 안 하는 API·함수 추측 사용 금지. 확인 못 하면 밝힌다.
7. 실패 시 원인 가설을 증거별로 분리 보고. 성공한 척 금지.
8. 라이브/DB/보안은 read-only 확인 먼저. 위험명령은 사용자 승인.
9. 막히면 멈추고 결과+막힌지점+시도+대안 보고.
10. 최종 보고는 변경점·검증결과·잔여위험 3가지만. "했다" 아니라 증거.

## 검증 프로토콜 (통합자 = 메인)

- 모든 산출물은 검증 대상(복붙 금지). 상관오류 경계 → 최종판단은 코드·테스트·로그 증거로.
- 검증을 "테스트 통과"로 축소 금지: 배포순서·데이터 마이그레이션·롤백 경로·운영 리스크도.
- 조기경보(그럴듯한 완료감): 테스트 미실행 · 안읽은 파일 단정 · 요구사항 일부 누락 · 큰 diff.

## 실패모드 / 조기경보

- 과분해 → 조율비용 폭증(사소한 건 직접).
- 통합자가 결국 가장 많은 맥락+가장 비싼 검증을 짊 → 판단만 한다는 목표와 충돌. 위임 덩어리를 충분히 크게.
- 추론기 자기 루프 장시간 자율 → 범위 팽창. 공짜라고 제한 풀지 말 것.
- 프로젝트별 규칙이 전역보다 늦게 적용되면 위험한 기본행동 선발생 → 프로젝트 규칙 우선.
