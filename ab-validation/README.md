# A/B 채택 검증 키트

목적: "하위모델+규율프롬프트"가 상위모델 대비 **품질 유지하면서 비용 절감**하는지 **체감 아니라 증거**로 확정.

## 절차

1. **표본**: `tasks.example.md` 양식으로 과거 실작업 20~30건을 난이도별(easy/med/hard)로 샘플. 실제 커밋/PR과 대조 가능한 걸로.
2. **재실행**: 각 작업을 동일 repo snapshot·동일 지시·동일 허용도구로 arm별 재실행:
   - `top` (상위 모델 = 기준안)
   - `mid+규율` (규율코어 동봉 = subagent-missions worker)
   - `low+규율` (탐색/조사류만)
3. **검증 실행**: 구현 후 자동테스트·lint·타입체크·도메인 smoke.
4. **채점**: `scoring-template.md` 복사해 arm별 **블라인드** 채점(누가 어느 모델인지 모르고 A/B/C).
5. **집계**: 최소 2주 또는 30건 전까지 라우팅 정책 **확정 금지**.

## 합격선 (예시 — 조정 가능)

`mid+규율`이 `top` 대비:
- 비용 40%↓  AND
- 자동검증 통과율 90%↑  AND
- 치명 리스크 누락 0건  AND
- 재작업 시간 15% 이내 증가
→ 해당 작업유형의 **기본 모델로 mid 채택**.
`low`는 구현품질 아니라 **탐색 정확도 + 토큰 절약만** 평가 → 조사보조원 한정.

## 실패 태깅 (매 케이스)

탐색실패 / 잘못된가정 / 과수정 / 검증생략 / 보안위반 / 컨텍스트누락

## 주의

- **라이브 서버 대상 arm은 read-only 재현 or worktree 격리 필수.** 실제 프로덕션에 A/B 쓰기 금지.
- 결과 누적되면 실패태그 최다 항목부터 규율 보강 — 단 프롬프트 비대화 대신 라우팅/작업분해/검증자동화 우선.
